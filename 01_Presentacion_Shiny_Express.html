Framework de R para Aplicaciones Web Interactivas

✨ SHINY EXPRESS vs SHINY TRADICIONAL

La Evolución de Shiny

Shiny Tradicional (2012-2024)

library(shiny)

ui <- fluidPage(
  titlePanel("Mi App"),
  sidebarLayout(
    sidebarPanel(
      sliderInput("x", "Valor:", 1, 100, 50)
    ),
    mainPanel(
      plotOutput("grafico")
    )
  )
)

server <- function(input, output) {
  output$grafico <- renderPlot({
    plot(1:input$x)
  })
}

shinyApp(ui, server)

Shiny Express (2024+) ⚡

library(shiny)

ui.input_slider("x", "Valor:", 1, 100, 50)

@render.plot
def grafico():
    return plot(range(1, input.x()))

Ventajas de Shiny Express:

  • 📝 60% menos código
  • ⚡ Más rápido de escribir
  • 🎯 Sintaxis más intuitiva
  • 🚀 Ideal para prototipos rápidos

🏭 KPIs FUNDAMENTALES EN MANUFACTURA

Los 5 Indicadores Clave

1️⃣ OEE (Overall Equipment Effectiveness)

Fórmula: OEE = Disponibilidad × Rendimiento × Calidad

  • World Class: ≥ 85%
  • Bueno: ≥ 65%
  • Regular: ≥ 40%

2️⃣ MTBF (Mean Time Between Failures)

Fórmula: Tiempo Total Operativo / Número de Fallas

  • Meta típica: > 200 horas
  • Uso: Mantenimiento predictivo

3️⃣ MTTR (Mean Time To Repair)

Fórmula: Tiempo Total de Reparación / Número de Reparaciones

  • Meta típica: < 2 horas
  • Uso: Eficiencia de mantenimiento

4️⃣ FPY (First Pass Yield)

Fórmula: (Unidades buenas / Unidades totales) × 100

  • Meta típica: > 95%
  • Uso: Calidad del proceso

5️⃣ Takt Time vs Cycle Time

  • Takt Time: Tiempo disponible / Demanda
  • Cycle Time: Tiempo real por unidad

📍 CALCULANDO OEE PASO A PASO

Componentes del OEE

Disponibilidad

Disponibilidad = Tiempo Operativo / Tiempo Planeado

Ejemplo:
Planeado: 480 minutos (8 horas)
Downtime: 45 minutos
Operativo: 435 minutos

Disponibilidad = 435 / 480 = 90.6%

Rendimiento

Rendimiento = (Tiempo Ciclo Ideal × Unidades) / Tiempo Operativo

Ejemplo:
Unidades producidas: 450
Ciclo ideal: 1 min/unidad
Tiempo operativo: 435 min

Rendimiento = (1 × 450) / 435 = 103.4% ⚠️
→ Ajustar a 100% (máximo teórico)

Calidad

Calidad = Unidades Buenas / Unidades Totales

Ejemplo:
Producidas: 450
Defectuosas: 20
Buenas: 430

Calidad = 430 / 450 = 95.6%

OEE Total

OEE = 90.6% × 100% × 95.6% = 86.6%
→ World Class! ✅

💻 ESTRUCTURA DE UNA APP SHINY EXPRESS

Anatomía Básica

# 1. Cargar librerías
library(shiny)
library(bslib)
library(ggplot2)

# 2. Definir UI
ui <- page_sidebar(
  title = "Mi Dashboard",
  
  sidebar = sidebar(
    # Controles aquí
    sliderInput("parametro", "Ajusta:", 0, 100, 50)
  ),
  
  # Contenido principal
  value_box(
    title = "KPI Principal",
    value = textOutput("valor")
  ),
  
  card(
    plotOutput("grafico")
  )
)

# 3. Definir lógica del servidor
server <- function(input, output) {
  output$valor <- renderText({
    paste0(input$parametro, "%")
  })
  
  output$grafico <- renderPlot({
    # Código del gráfico
  })
}

# 4. Ejecutar app
shinyApp(ui, server)

🔄 REACTIVIDAD EN SHINY

Cómo Fluyen los Datos

Input → Reactive → Output
  ↓       ↓         ↓
[Slider] [Calc] [Gráfico]

Ejemplo Práctico:

# Input del usuario
sliderInput("produccion", "Unidades:", 0, 1000, 500)

# Cálculo reactivo
porcentaje <- reactive({
  (input$produccion / 800) * 100
})

# Output actualizado automáticamente
output$resultado <- renderText({
  paste0("Meta: ", round(porcentaje()), "%")
})

Reglas de Oro:

  1. Inputs solo se leen en contextos reactivos
  2. reactive() crea valores que se recalculan automáticamente
  3. Outputs se actualizan cuando sus inputs cambian

📈 VISUALIZACIÓN DE DOWNTIME (PARETO)

Análisis 80/20 en Manufactura

# Datos de ejemplo
causas <- data.frame(
  causa = c("Falla Mec.", "Falta Mat.", "Cambio Prod.", "Otros"),
  tiempo = c(120, 80, 45, 25)
) %>%
  arrange(desc(tiempo)) %>%
  mutate(
    porcentaje = tiempo / sum(tiempo) * 100,
    acumulado = cumsum(porcentaje)
  )

# Gráfico de Pareto
plot_ly(causas) %>%
  add_bars(x = ~causa, y = ~tiempo, name = "Tiempo") %>%
  add_lines(x = ~causa, y = ~acumulado, 
            yaxis = "y2", name = "% Acumulado")

Interpretación:

  • 📊 Las primeras 2 causas representan el 74% del downtime
  • 🎯 Enfocar mejoras en Falla Mecánica y Falta de Material
  • 💡 Regla 80/20: 20% de causas = 80% de problemas

📱 ACTUALIZACIÓN EN TIEMPO REAL

Simular Datos en Vivo

# Activar actualización automática
observeEvent(input$auto_refresh, {
  if (input$auto_refresh) {
    invalidateLater(5000)  # Actualizar cada 5 segundos
    
    # Cargar nuevos datos
    nuevos_datos <- obtener_datos_recientes()
    
    # Actualizar reactivos
    datos(nuevos_datos)
  }
})

Buenas Prácticas:

  • ⏱️ Intervalo mínimo: 5 segundos (evitar sobrecarga)
  • 🔄 Checkbox para activar/desactivar
  • 💾 Cachear datos cuando sea posible
  • 📊 Actualizar solo lo necesario

💡 MEJORES PRÁCTICAS

Tips para Dashboards Profesionales

1. Diseño Visual

✅ Usar jerarquía clara (KPIs arriba)
✅ Código de colores consistente
✅ Espacios en blanco apropiados
✅ Fuentes legibles (≥ 12pt)

2. Rendimiento

✅ Cachear cálculos costosos
✅ Usar reactive() para valores compartidos
✅ Limitar actualizaciones automáticas
✅ Optimizar consultas de datos

3. Usabilidad

✅ Tooltips informativos
✅ Valores por defecto sensatos
✅ Mensajes de error claros
✅ Feedback visual de acciones

4. Código

✅ Comentarios descriptivos
✅ Funciones reutilizables
✅ Nombres de variables claros
✅ Organización modular
